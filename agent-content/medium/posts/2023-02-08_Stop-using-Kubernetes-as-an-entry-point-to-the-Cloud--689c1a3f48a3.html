Stop using Kubernetes as an entry point to the Cloud…

I am aware that this viewpoint may be controversial. Nonetheless, based on my 20 years of experience working with customers across different technologies and organizations of varying maturity, I can say that Kubernetes is a fantastic, flexible, extensible, and impressive technology.

It is also complex, and its management overhead can often diminish the flexibility that is a key aspect of the platform.

There are many very good reasons for using the cloud:

  1. Reducing undifferentiated heavy lifting — Using the Cloud allows you to shift your attention away from racking and stacking servers. You can also stop running networks and focus on delivering key differentiations. This should includes things like deploying applications and features that end users can consume.
  2. Reducing the time to market — Getting new features shipped should be much quicker as you eliminate unneceessary heavy lifting. Automation becomes a breeze when using a cloud environment.
  3. Reducing operating overheads — Moving to primitives (such as serverless or SaaS platforms) abstract the underlying technologies.
  4. Variable Cost — Scale cost up and down as needed in a dynamic way.

Most of these cost savings can be quickly offset if the standard approach is to use Kubernetes for all cloud workloads. It’s important to note that there are valid use cases for Kubernetes, but it should not be the default starting point.

What are some other alternatives to Kubernetes?

There are various approaches that can be used in the cloud. It’s advisable to consider the following alternatives and eliminate them before considering Kubernetes:

  1. SaaS solutions — is there a better alternative which can be provided using software as a service?
  2. PaaS solutions — Platforms as a service options will often give you the desired outcomes without needing to manage the underlying infrastructure
  3. Serverless — Consider approaches such a serverless primatives (eg, State Machines, Functions as a Service). If you can abstract the underlying infrastructure and only focus on the business logic then you will still leverage the benefits of the cloud without the complexity of Kubernetes
  4. Other Contain Orchestration mechanisms — Kubernetes is the beast with all the bells and whistles. But are you actually going to use them? For example, on AWS there are 17 different ways to run continers — and only 1 of them is based on Kubernetes
  5. Use Kurbenetes as a managed service — All of the hyperscalers have a Kubernetes flavour which at least has a bonus control plan. Bonus points for using a serverless control data plan.

If you have exhausted all other options and still cannot achieve what you need, you can consider building and operating a Kubernetes cluster. However, be aware that it is not a quick process. It will require a substantial amount of effort to deploy and there will be ongoing maintenance and support requirements.
