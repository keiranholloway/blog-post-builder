An Introduction to Event Driven Architecture…

Cloud technologies introduce a wide array of new ways to accomplish tasks and deploy solutions at scale. One of the most interesting architectural patterns is event-driven architecture (EDA) practices, which serve as an effective way to disaggregate the traditional monolithic application, as well as provide an extensible data plane for microservices to consume inputs. This blog post is intended to provide an introduction to EDA, as well as a high-level example of how it could be implemented.

Event-driven architecture is a software design approach that allows different components of a system to communicate with each other by producing and consuming events. An event is a significant occurrence, such as a user’s action, a change in the system’s state, or an external system’s notification. When an event occurs, the appropriate component(s) of the system will respond by processing the event and taking appropriate action.

Event Driven Architecture Components…

At the highest level, there are effectively 4 components:

1. Events: These are the signals that indicate that something has happened in the system. Events can be generated by users, applications, or other systems.

2. Event Message Bus: An event message bus, also known as an event bus or event streaming platform, is a messaging infrastructure that enables event-driven communication between different components of a distributed system. Events are typically published to an event message bus.

3. Event producers: These are the components that generate events. They can be any application, system, or device that can generate an event which adds the event to the messaging bus.

4. Event consumers: These are the components that consume events. They can be any application, system, or device that needs to react to an event. These will read messages off the Event messaging bus.

Common Event Driven Architecture Patterns…

There are a range of common event driven architecture patterns including:

1. Event notification pattern: This pattern is used when one component needs to inform other components that a particular event has occurred. The component producing the event publishes it to an event broker, and the interested subscribers can consume the event to take necessary action.

2. Event-driven messaging pattern: This pattern involves passing messages between different components of the system to trigger a response. The message contains information about the event that has occurred, and the recipient processes it accordingly.

3. Command Query Responsibility Segregation (CQRS) pattern: This pattern separates the read and write operations of the system into separate components, known as commands and queries. Commands are responsible for updating the system state and publishing events, while queries are responsible for retrieving the system’s state.

4. Event sourcing pattern: This pattern involves storing the entire history of events that have occurred in the system, rather than just the current state. This approach allows the system to be reconstructed at any point in time, which can be useful for debugging, auditing, and compliance purposes.

5. Saga pattern: This pattern is used to manage long-running transactions that involve multiple components. It uses a set of coordinated compensating actions to ensure that the transaction can be rolled back if any component fails.

Benefits of using Event Driven Architecture…

There are several reasons why you might want to use an event-driven architecture (EDA) for your software design:

1. Scalability: EDA can help you build highly scalable systems that can handle a large volume of events and users. By allowing different components to operate independently and asynchronously, you can distribute the load across multiple servers or nodes.

2. Flexibility: EDA allows you to add or remove components from the system without affecting the other components. This can make it easier to modify and evolve the system over time.

3. Decoupling: EDA can help you decouple different components of the system, so that they don’t have to know about each other’s implementation details. This can make the system more modular and easier to maintain.

4. Real-time processing: EDA allows you to process events in real-time, which can be useful in applications where quick responses are required.

5. Event-driven workflows: EDA can help you design workflows that are triggered by events, such as processing a payment or sending a notification. This can help you automate business processes and improve efficiency.

Event Driven Architect Example…

In this example, we are focused on Bank or similar type of financial organisation. The following depicts the flow of events using event driven architecture. The flow is from left to right.

On the left hand side (Event Producers) there is a system of record (SoR) which is generating transactions and publishing events to the Event Message Bus. The types of messages which would be published could cover a number of different scenarios including. For example:

1. Money successfully deposited 
2. Successful withdrawal from an account 
3. Transaction challenge (for example, validating that this is a real account).

The event will be generated as result of a specific acation. The system of record will then publish this transaction activity to the event message bus with sufficient metadata around the transaction (for example, success or failure).

Microservices can then consume these messages off the queue in a completely decoupled way. All the business logic is self-contained within each microservice which means that these can be added, removed or changed independently of other components within the application. This approach allows for end-to-end transaction, in a decoupled way.

Some examples…

Transaction One: A successful withdrawal of $200 from a bank account

  1. The system of record approves the transaction and publishes an event to the message bus recording this transaction.
  2. A fraud detection service can analyze the transaction out of band to characterize the likelihood frequently and notifying the bank if it looks suspect.
  3. Simultaneously a push notification can be sent to a mobile phone app notifying the customer of successful completion of the transaction
  4. A record of this transaction can be sent to a database warehousing service which can then generate end of month summaries of spending.

All of this can happen concurrently and in a de-coupled way. Services can be scaled independently, changes can be rolled out and only affect the single microservice which is being updated.

Transaction two: A customer requests a withdrawal of $3000 which the SoR is deemed to require further authorization…

In this example, the SoR gets a request to transfer a large amount of money. Business logic has been added to the SoR to ensure that this has secondary approval before releasing the funds.

In this example, the same Event Message Bus can be used but with different outcomes. The format of the event can indicate that this transaction is actually “In Progress” versus “Complete”. In effect, the Microservices could take the following actions:

  1. Fraud Detection will review the transaction to see if it has characteristics of scam and notify the bank security team in real-time if this looks suspicious. This could prevent it from being completed.
  2. The Push notification app can send a message highlight that this has been requested — but not approved. The end user could take action if this is not expected.
  3. Data warehouse will ignore this event — as it has not successfully completed yet.
  4. The OPT Auth application though will generate the one time password which is sent to the customer out-of-band to confirm if this is a legitimate request before proceeding with the transaction.

Overall, event-driven architecture can help you build more robust, scalable, and flexible systems that can respond to events and changes in the environment in a timely and efficient manner.
