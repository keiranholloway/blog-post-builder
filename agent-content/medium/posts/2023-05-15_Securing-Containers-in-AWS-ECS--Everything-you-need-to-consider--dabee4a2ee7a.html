Securing Containers in AWS ECS: Everything you need to consider…

AWS provides a lot of different ways to run containers. In fact, listing all the ways in which AWS let you a container is not an easy feat! For now though, we’re going to focus on one: Amazon Web Services: Elastic Container Service (ECS).

Securing Containers on ECS…

This post comprehensively covers what you need to think about when it comes to running containers within ECS. This intended to discuss security aspects from the bottom up covering:

  • AWS ECS control plane
  • Underlying Infrastructure
  • Worker nodes (with both EC2 and Fargate)
  • Network Security
  • Identity and access management
  • Security best-practices within the container itself
  • CI/CD tooling and best practices using a DevSecOps approach

Control Plane Security

Securing the control plane is crucial for maintaining a secure container environment. Some best practices include:

  • Enabling AWS CloudTrail for logging and monitoring API calls made within your AWS environment.
  • Using AWS Config to continuously assess, audit, and evaluate the configurations of your AWS resources. There are a number of AWS managed config rules for ECS which you can enable relatively quickly to identify areas where security can be enhanced.
  • You can also look at limiting which IP addresses can successfully make calls to the AWS ECS API. See conditional policies which limit access based on source IP address. Be aware of automated processes which may need to use this API (for example, autoscaling). Test this a lot in lower environments before promoting to production.

Infrastructure Security

Securing the underlying infrastructure is essential for the protection of your containers. Consider the following practices:

  • Regularly patching and updating your operating systems running your worker nodes and any associated software.
  • AWS Security Hub to monitor and manage security alerts and AWS Config automate compliance checks.
  • Using Amazon GuardDuty for intelligent threat detection and continuous monitoring.

Worker Nodes: Fargate vs. ECS

General operational hygiene should be observed within the cloud. This includes things like:

  • Do not hard code credentials where not necessary. This means using primitive such as IAM Roles and AWS services such as AWS Secrets manager or SSM parameter store.
  • Ensure that credentials are not long-lived. This means that they should be periodically rotated.
  • Use secure networking mode. I would recommend using ‘awsvpc’ which means that ECS creates and managed a separate ENI for each task. This means every task will get a unique private IP address within the VPC which can have security groups directly applied and provide enhances security at the container level.

Both Fargate and EC2 launch types have different shared responsibility models. It is critical to understand your roles and responsibilities under each of these models.

EC2 Worker nodes

This could not be considered a comprehensive article of ECS security without the obligatory picture of the share responsibility model. It is very important to note though, that scope of responsibility does shift between EC2 workers and Fargate.

You will note that the responsibilities with EC2 focus on the virtual machine host itself. Areas which you need to consider:

  • The IAM role which is attached to the EC2 instance should be as restricted as possible. Any task which runs on a worker node can inherent these permissions so this should be very limited. You can then grant permissions are required on a container basis using IAM roles defined in the task definition. This will override the EC2 IAM role.
  • Use an AMI which is provide enhanced security. The ECS Optimized AMI from AWS is a good starting point. There are other more security focused AMIs available too. For example: Bottlerocket
  • Keeping the AMI is your responsibility too when using EC2. It is recommended that you have a mechanism in place to update your images on-boot or frequently replace images with updates applied. I am a strong advocate of automating this.

Fargate Worker nodes

Alternatively, with fargate the shared responsibility model shifts slightly with the worker node responsibility shifting to AWS. As result, this provides a higher level of isolation between tasks. Some of the best-practice which should be followed include:

  • Limiting the permissions of the task role to the minimum required. This means that fine-grained permissions should be defined and applied that mature the requirements of the container code.
  • Ensure that you are running the latest fargate platform version. This may mean periodically recycling the task and ensuring that you specify the latest platform version.

Network Security

Implementing network security measures helps protect your containers from potential threats. Some best practices include:

  • Do not assign public/internet routable address space to tasks. Your ECS worker nodes should be sitting in private subnets only and using ‘awsvpc’ network mode. This means that every task will receive its own elastic network interface and private IP addresses. Be aware that small EC2 instance are limited to how many ENIs they can have attached and make sure you have enough free IP addresses within your CIDR network segment.
  • With your ECS worker nodes sitting in private segments services should only be exposed publicly sitting behind a network intermediary (for example, an Elastic Load Balancer). Tasks should never be directly exposed to the public Internet.
  • Restrictive security groups in conjunction with network ACLs should be used for limiting traffic and protecting your containers.
  • Employing AWS WAF (Web Application Firewall) to protect against common web exploits is recommended. There are also other third-party layer 7 firewalls which provide enhanced security. For example, Akamai’s Kona site defender, Imperva’s FlexProtect or CloudFlare’s WAF products.

IAM Approaches

Using Identity and Access Management (IAM) effectively is crucial for safeguarding your environment. There are a range of general best practices which need to be considered:

  • Implementing the principle of least privilege for IAM roles and policies. This should consider worker nodes and permissions which are applied to containers.
  • Regularly auditing IAM roles, users, and groups for compliance. Users really should only be used of real humans signing into the environment. IAM roles should be applied to routine changes — your CI/CD Pipelines should be authenticating using OpenID Connect to avoid the need for long-live credentials to be hard-coded anywhere.
  • Enabling multi-factor authentication (MFA) for all IAM users.
  • Use task execution roles for every task/container. This means that you can use very restrictive IAM Roles on the worker nodes and then have fine grain controlled applied at the container/task level.

Container Security

To ensure security within the container, consider the following practices:

  • Utilizing trusted base images and scanning images for vulnerabilities. Amazon’s ECS provives both basic and advanced scanning. It is recommended that you use Amazon Inspector which can feed observed vulnerabilities to AWS EventBridge. From here you can be notified of events and work towards remediation.
  • Implementing runtime security measures, such as AppArmor or SELinux.
  • Limiting resource usage to prevent resource exhaustion attacks. This is particularly important when it comes to CPU or Memory limits; ensure you are using hard limits.
  • Containers must be running in non-privilege mode. This includes running the container as a user other than root.
  • Ensure that you task definitions should not share the host’s process namespace.
  • Tasks/Containers should be limited to read-only access to root filesystem. This protects against attacks where malicious code is somehow uploaded or injected into a container and then executed.
  • Secrets should not be passed as container environment variables.
  • Ensure that metrics and logs are sent from the containers to a common logging platform. Examples of tools include: AWS Container Insights, CloudWatch Logs or third-party tools like Datadog or Splunk.
  • If you are using ephemeral or persistent storage — ensure that this is encrypted.
  • Ensure that unencrypted and other insecure ports are not exposed from the container unnecessarily.

DevSecOps and CI/CD Tooling

Incorporating DevSecOps principles into your CI/CD pipelines can help identify and mitigate security risks early in the development process. Identifying issues and remediating them in lower environment (shifting-left) is always a good approach to save you time.

Some examples of tools and practices include:

  • Static application security testing (SAST) tools like SonarQube , Fortify or Snyk
  • Dynamic application security testing (DAST) tools like OWASP ZAP or Burp Suite.
  • There are also open source container image scanning tools like Clair or Trivy.
  • Infrastructure as Code (IaC) security scanning tools like Checkov or Terrascan to validate your IaC repositories to mitigate risks.
  • Employing vulnerability and dependency scanning tools like Snyk or WhiteSource.
  • Security scanning with tools like Cycode or enabling secret scanning within your code repository tooling (Eg, Github Security scanning).

Conclusion

Containers provide a huge number of benefits and can speed up your organization. Maintaining the security of this infrastructure needs to remain at the forefront. Hopefully this article has provided a number of approaches on how to address this.
